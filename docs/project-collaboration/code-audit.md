## 已在本轮修复的问题

### 1. README 与真实入口不一致

问题：

- 旧 README 使用了不存在的 `config/risk_model.yaml`
- 训练和评估命令没有反映当前代码真实支持的任务模式

处理：

- 已按当前仓库真实入口改写 README
- 明确 `config/settings.py` 才是默认配置中心
- 补齐分类、回归、多模态三类命令示例

### 2. `train.py` 分类测试阶段把 logits 当成预测类别

问题：

- `scripts/train.py` 在测试集阶段直接收集 `outputs.cpu().numpy()`
- 后续 `classification_report` 需要的是类别索引，不是原始 logits

影响：

- 分类任务下测试集报告会失真，甚至直接报格式错误

处理：

- 已改成先 `argmax` 再统计分类结果

### 3. 推理端不能直接加载训练得到的回归 / 多模态 checkpoint

问题：

- `src/inference/predictor.py` 之前只能识别到 LSTM / 多模态 checkpoint 类型
- 但没有实际重建模型并加载权重

影响：

- 训练与推理链路不闭合
- `demo.py` 即使提供 `.pt` checkpoint，也可能无法真正用于风险预测

处理：

- 已补齐回归 / 多模态 checkpoint 的自动重建与加载逻辑
- 推理时可以直接对关键点序列做评分

### 4. 评估入口与训练入口任务模式不对齐

问题：

- 训练入口支持 `classification` / `regression` / `multimodal`
- 原评估脚本基本只覆盖分类路线

处理：

- 已补齐评估脚本的 `--task` 入口
- 让模型构建、数据集读取、指标计算与训练任务对齐

## 当前仍然存在、但未在本轮彻底解决的问题

### 1. 风险评分标签目前偏“代理标注”

位置：`src/data/dataset.py`

情况：

- 当前连续风险分数大量由动作语义规则映射得到
- 更像研究阶段代理目标，而不是真实临床风险标签

建议：

- 后续单独整理风险标注方案，区分“动作类别标签”和“风险评分标签”来源

### 2. 多模态环境特征仍以占位向量为主

位置：

- `src/data/dataset.py`
- `src/models/multimodal_risk.py`

情况：

- 多模态框架已经存在
- 但当前环境 / 轨迹分支在实际训练数据流里仍缺少真实特征输入

建议：

- 先明确环境特征定义，再把真实数据管道接进训练与推理

### 3. HRNet 后端尚未真正实现

位置：`src/models/pose_estimation.py`

情况：

- MediaPipe 可用
- HRNet 仍是占位接口

建议：

- 如果比赛或实验需要高精度离线姿态，可优先补这一块

### 4. 数据下载脚本仍是说明型脚本

位置：`scripts/download_data.py`

情况：

- 当前脚本主要负责告诉用户去哪下载、放到哪里
- 还不是真正的一键下载脚本

### 5. 自动化测试覆盖情况仍不清晰

情况：

- 仓库存在 `tests/` 目录
- 但本轮未通过目录接口完整展开其内容
- 当前也还没有看到显式 CI 配置

建议：

- 后续单开一轮，把测试目录和回归检查链路单独梳理清楚

## 结论

当前仓库已经具备较完整的研究原型主线，但训练、评估、推理、文档之间此前确实存在几处关键错位。本轮优先修复的是会直接影响“命令能不能跑通、checkpoint 能不能接上、评估结果准不准”的问题。剩下最需要继续投入的，是更真实的风险标注、多模态真实特征接入，以及更稳定的测试与实验规范。
