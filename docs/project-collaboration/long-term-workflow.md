# 老人跌倒风险预测项目长期协作方案

## 目标

建立一套可重复访问、可持续扩展、适合长期共同修改项目的工作方式，减少重复沟通成本，并保证论文理解、代码分析、实验设计和仓库改动能持续对齐。

## 默认适用范围

- 默认仓库：`yanzhaoing/fall-risk-project`
- 默认任务类型：
  - 仓库结构理解
  - 论文到代码映射
  - 实验设计与对比分析
  - bug 排查
  - README / 技术文档整理
  - 在明确目标下进行最小必要修改

## 长期协作节奏

### 1. 每次开始时

1. 明确这次任务属于哪一类：
   - 代码理解
   - 论文分析
   - 论文到代码映射
   - 实验设计
   - bug 排查
   - 仓库修改
   - 文档整理
2. 只读取当前任务真正相关的仓库文件、论文或配置。
3. 先给出定位和判断，再决定是否修改代码或文档。

### 2. 每次分析时

统一遵循下面的输出框架：

1. 当前目标是什么。
2. 相关代码 / 配置 / 论文 / 数据流在哪里。
3. 已确认的事实是什么。
4. 仍不确定的点是什么。
5. 下一步最小可执行动作是什么。

### 3. 每次修改前

统一先确认四件事：

1. 要改什么。
2. 为什么改。
3. 影响范围在哪。
4. 怎样验证改动是否成立。

### 4. 每次修改后

统一补充四类信息：

1. 修改摘要
2. 受影响文件
3. 验证方式或尚未验证点
4. 后续建议

## 推荐的长期记录单元

### A. 项目档案

用于保存稳定信息：

- 仓库名
- 项目目标
- 关键模块
- 常用脚本
- 已知依赖
- 论文与代码的主映射

### B. 会话记录

用于保存每次任务的可追溯结果：

- 日期
- 本次目标
- 查看了哪些文件 / 论文
- 得到什么结论
- 是否有改动
- 后续待办

### C. 决策记录

当项目出现重要选择时单独记录：

- 为什么采用该方案
- 替代方案是什么
- 风险是什么
- 以后什么条件下要重新评估

## 我们之后的默认配合方式

1. 先读上下文，再给结论。
2. 如果只是分析、对照、排查，我不主动改仓库。
3. 如果要改仓库，我会先缩小改动范围，再做最小必要修改。
4. 每次结束时，把有长期价值的信息继续沉淀到仓库文档里，方便后续直接续接。
